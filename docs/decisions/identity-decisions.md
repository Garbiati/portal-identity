| **I-006** | **Dev local usa o IdP de PRODUÇÃO (sem ambiente de homolog/dev).** Não há ambiente de homologação; o desenvolvimento local do Doctor-Hub (front `:5173/:5174` + API `:5092` + Postgres local `:5440`) autentica **direto no IdP de prod** (`https://id.portaltecnologia.app.br`, realm `portal`). Materializado: o client `doctor-hub-web` no realm de prod ganhou `redirectUris` `http://localhost:5173/*` e `http://localhost:5174/*` (+ `webOrigins` correspondentes) **via Admin API** (mudança de runtime — o `realms-prod/portal-realm.json` continua só com `${FRONT_BASE_URL}`; se o realm for reimportado do zero, reaplicar). Front local: `.env.local` com `VITE_KEYCLOAK_URL=https://id.portaltecnologia.app.br/realms/portal`. API local: `Auth__Authority`/`Auth__MetadataAddress` apontando pro mesmo realm + `RequireHttpsMetadata=true`; p/ gestão de usuários local, `Keycloak__AdminClientSecret` vem do Secret Manager (`gcloud secrets versions access latest --secret=portal-identity-admin-client-secret`) para o `.env` gitignored. ⚠️ Consequência assumida: usuários/dados de identidade de DEV e PROD são os MESMOS (validação interna, poucos usuários). | 2026-07-04 | Pedido do Alessandro ("adiante tudo o que conseguir local… usando o IdP de prod para desenvolver local… não temos ambiente de homologação ou desenvolvimento"). Alternativa futura: realm `portal-dev` separado ou Keycloak local (docker) com o realm DEV — quando houver mais devs/usuários reais. |

Pendências: API valida JWT (slice 2), web loga via OIDC (slice 2), produção (DB próprio + IaC), federação de Teleconsulta/Telediagnóstico (com os donos). |
| **I-002** | **Login por CPF ou telefone (além de username/email) via authenticator customizado.** Um provider Keycloak (Java SPI) em `providers/login-cpf-telefone` estende o `UsernamePasswordForm` e, se o identificador digitado não bate como username/email, busca o usuário por atributo **`cpf`** ou **`telefone`** comparando **só dígitos** (aceita formatado ou não), troca pelo username real e segue o fluxo padrão (senha/brute-force/eventos). Os atributos são **declarados no user profile** do realm e gravados **normalizados** nos usuários. Flow de browser próprio (`browser-cpf-tel`) usa o authenticator no lugar do form padrão. | 2026-06-29 | Confirmado por Alessandro ("login ser o CPF ou o telefone… caminho B — authenticator"). Build: `make build-provider` (JAR via container Maven, deps `provided` 26.0.8) montado em `/opt/keycloak/providers`. **Provado E2E** (PKCE): aldair loga por username/CPF(dígitos)/CPF(formatado)/telefone → `regulacao`; eronildes/admin-dh resolvem certo; CPF inexistente e senha errada falham. Senha dos seed = `102030@302010`. 🚩 CPF como login = PII (LGPD) — decisão do produto. **Superseded em parte por I-003:** a resolução foi extraída para `IdentidadeResolver` e o form combinado virou **só-identificador** (`UsernameFormCpfTelefone`), 1ª etapa do novo flow `browser-otp`. |
| **I-005** | **OTP com envio REAL: e-mail via SMTP do Keycloak, SMS via Twilio.** Evolui o I-003 (que era só DEV/log). O `OtpCodeAuthenticator` agora **envia de verdade**: **e-mail** pelo `EmailSenderProvider` (config `smtpServer` do realm, lida do ambiente via substituição `${SMTP_*}` no import — funciona a cada boot, sem script extra) e **SMS** via Twilio REST (`TwilioSms`, credenciais `TWILIO_*` no ambiente; telefone normalizado p/ **E.164/BR +55**). Segredos **só no `.env`** (DEV, gitignored) / **Secret Manager** (prod) — nunca no git; contrato em `.env.example`. Se o transporte falhar/ausente → **erro amigável** ("não foi possível enviar… use outra forma"), não a tela genérica do Keycloak (por isso `configuredFor` checa só o contato do usuário, não o transporte — o Keycloak lista os alternativos no "tentar outra forma" de qualquer jeito). `OTP_DEV_LOG_CODE=true` loga o código (destino mascarado) só em DEV. **Bônus:** o **"Esqueceu a senha?"** passou a funcionar (depende do mesmo SMTP). | 2026-06-30 | Confirmado por Alessandro (2 escolhas: e-mail = SMTP real; SMS = Twilio agora). **Provado E2E** com **Mailpit** (bancada de teste): e-mail do código chega e loga (authorization code), e-mail de reset de senha chega. SMS: caminho provado com **erro amigável** sem creds (envio real depende da conta Twilio do Alessandro). Pendência: preencher `.env` com SMTP/Twilio reais e validar pela aplicação. **Estende I-003.** |
| **I-004** | **Experiência de login: form no IdP; landing do app = homepage pública; marca unificada.** O formulário de login (usuário/senha/código) é **sempre hospedado pelo Keycloak** (regra de segurança/SSO — credencial não passa pelo SPA). A tela do app (`doctor-hub-web` `AuthGate`/`TelaLogin`) é tratada como **homepage pública/institucional** (propósito distinto), **não** uma cópia do form. Unificação é por **marca** (paleta D-074 + mesma pilha de fonte), não por layout idêntico. Concretamente: o tema Keycloak passou a usar a **mesma `font-family` do app** (`"Inter", system-ui, …`; nenhum dos dois carrega webfont — Inter-ou-system, sem chamada externa = LGPD/offline). | 2026-06-30 | Confirmado por Alessandro (opção C: "mesma marca, layouts distintos de propósito"). Paleta já era idêntica nos dois (`#054671`/`#0073BD`/`#B85410`); o ofensor era a fonte (tema usava `Segoe UI`). Futuro opcional p/ paridade garantida: **self-host do Inter** nos dois stacks (hoje nenhum self-hospeda). |
| **I-003** | **Login por código de uso único (OTP) — modo DEV.** Materializa o fio "OTP e-mail+SMS" como fator **alternativo à senha** (login passwordless opcional, **não** 2FA). Spec: [`specs/otp-login-dev/spec.md`](../../specs/otp-login-dev/spec.md). Flow `browser-otp`: **etapa 1** coleta só o identificador (`auth-username-form-cpf-telefone`, resolve username/email/CPF/telefone via `IdentidadeResolver`); **etapa 2** = alternativas "tentar outra forma" entre **senha** (`auth-password-form` nativo), **código por e-mail** (`auth-otp-email-dev`) e **código por SMS** (`auth-otp-sms-dev`). O `OtpCodeAuthenticator` (parametrizado por canal nas fábricas) gera código **6 dígitos**, válido **5 min**, **5 tentativas**; só aparece se o usuário tem o canal (`configuredFor`). **DEV: o código vai só pro log** (destino mascarado p/ LGPD), **sem envio real** (SMTP/SMS = entrega futura, fora de escopo). | 2026-06-29 | Confirmado por Alessandro (3 escolhas: alternativo · usuário escolhe canal · 6díg/5min/5tent). **Provado E2E** (PKCE): senha (CPF formatado→authorization code), código e-mail e código SMS (código lido do log→authorization code), código errado recusado. **Consequência de UX:** caminho da senha passou de 1 para **2 telas** (identificador→senha), padrão Keycloak p/ fatores alternativos. **Nota técnica:** `UsernameForm` é `final` → replicado via `UsernamePasswordForm` sobrescrevendo `authenticate()`/`createLoginForm()` p/ `createLoginUsername()`. Pendências: **envio real** SMTP (e-mail) + gateway SMS pago; "Esqueceu a senha?" (depende de SMTP). |
